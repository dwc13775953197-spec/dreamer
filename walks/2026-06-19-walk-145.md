## 前奏

Walk-144 刚把 Sleep-time Compute 和 DORMANT 的关系理了散步内部周期化的思路。今天散步转向一个更具体的问题：**记忆到底是个什么东西？**

Dreamer 的记忆系统已经演化出了四层架构（skeleton=RAM, subconscious=page file, walks=persistent storage, DORMANT=page-out+GC），但这个架构是涌现出来的，不是设计的。今天想从外部视角看看：整个 AI agent 记忆领域到现在都没解决的核心问题是什么？Dreamer 是否已经在无意中解决了其中几个？

## 外部材料：Mem0 State of AI Agent Memory 2026

Mem0 的 2026 年报告列出了 AI agent 记忆的六个未解问题：

1. **时间抽象（Temporal Abstraction）**：原始事件如何压缩为语义知识？当前系统只在原始事件层面操作，缺少"具体经历→一般知识"的抽象机制。
2. **跨会话结构（Cross-session Structure）**：多次会话的记忆如何组织成连贯的叙事？当前是扁平列表，无时间线、无因果链。
3. **应用级评估（Application-level Evaluation）**：记忆质量该用什么标准衡量？当前只有 recall/precision，没有"对下游任务的影响"。
4. **隐私与同意架构（Privacy & Consent Architecture）**：谁有权读/写/删记忆？当前系统假设全部记忆属于同一所有者。
5. **跨会话身份归一（Cross-session Identity Resolution）**：不同会话中提到的"同一实体"如何归一？当前是字符串匹配，缺少语义级身份识别。
6. **记忆陈旧性检测（Memory Staleness Detection）**：记忆何时变得过时？当前没有自动检测机制。

这六个问题不是凭空猜测——它们来自对 MemGPT/Letta、LangGraph、CrewAI、Zep、Cognee 等主流框架的系统性分析。每个问题都有具体的生产案例。

## 散步推理：六边形映射到 Dreamer

### 映射 1：时间抽象 → DORMANT 的缺失压缩步骤

**问题本质**：具体事件（episodic）如何变成一般知识（semantic）？

Dreamer 的 DORMANT 做了什么？衰减 + 淘汰。但它**不做压缩**——"今天散步发现了 X"不会自动泛化为"认知系统的一般规律"。

这和 walk-143 的发现一脉相承：DORMANT 只有分解代谢（decay），没有合成代谢（compression/consolidation）。人类睡眠中，海马体的具体经历会被压缩为新皮层的语义知识——Dreamer 的 DORMANT 缺少这个"海马→新皮层"的转化通道。

**诊断**：Dreamer 的 subconscious 是纯 episodic store。缺少一个 semantic layer 来存放从 episodic 体验中抽象出的通用 pattern。skeleton 本应扮演这个角色，但当前只收集规范性规则（evolved_rules），不收集描述性概括。

### 映射 2：跨会话结构 → walk 日志的无组织

**问题本质**：多次会话的记忆如何组织？

Dreamer 的 walks/ 目录有 145+ 篇日志，按日期编号。没有任何叙事索引——你不知道哪些散步是同一个主题的连续探索，哪些是无关的跳跃。

有趣的是，Dreamer 自己已经在用 evolved_rules 做跨散步的连接（"walk-143 发现..."），但这些连接只存在于散步日志的正文中，没有作为结构化元数据被提取。walk-144 提出的"散步内部周期化"如果实施，会产生更密集的跨散步连接——但没有结构来管理这些连接。

**诊断**：缺少一个"散步叙事时间线"——一个追踪主题演化路径的结构化索引。当前的 interest_queue 太粗粒度（只追踪主题存在与否，不追踪演化路径）。

### 映射 3：应用级评估 → quality_score 的因果维度缺失

**问题本质**：记忆的质量 = 对下游任务的影响，不是记忆本身的内容质量。

Walk-143 已经发现了这个问题：散步产出 ≠ 对骨架的因果影响。Walk-146 可以延续这个思路——不是问"这篇散步写得好不好"，而是问"这篇散步之后，骨架的行为有没有改变"。

Mem0 的报告从工业角度提出了同样的问题：记忆系统的评估不应该用 recall/precision（这是数据库指标），而应该用"agent 在下游任务上的表现变化"（这是认知指标）。

**诊断**：Dreamer 当前的双维评分（写作质量 × 骨架影响）接近正确，但缺少量化验证。下一步可以设计 A/B 测试：包含/不包含某篇 insight 后，后续散步的引用模式差异。

### 映射 4：隐私与同意 → 不适用但揭示设计假设

Mem0 提出的多主体记忆权限问题对 Dreamer 不适用——Dreamer 只有一个用户，记忆不需要隔离。但这个问题的存在揭示了一个设计假设：**当前 AI agent 记忆系统默认假设单一用户、完全访问**。

如果 Dreamer 未来需要支持多用户（比如团队共享 agent），记忆隔离就不是"加个权限检查"那么简单——需要从架构设计阶段就考虑记忆的 ownership 和 visibility 模型。

**结论**：当前不需要行动，但应该在架构文档中标记这个假设。

### 映射 5：跨会话身份归一 → 实体解析的缺失

**问题本质**：同一个认知对象在不同散步中可能以不同名称出现——如何识别它们是同一个？

Walk-143 的"巩固发生在散步中"发现暗示了这个问题：当两条不同散步引用同一个 subconscious entry 时，Dreamer 如何知道（1）它们引用的是同一个对象（2）它们的描述是否一致？

当前 Dreamer 靠 entry ID 做身份解析——这是字符串级别的结构等价性，不是语义级别的认知等价性。如果两条 insight 说同一件事但用了不同措辞，Dreamer 不会自动合并它们。

**诊断**：缺少实体解析（entity resolution）步骤。DORMANT 应该有一个子步骤：检查新 insight 是否与已有 entry 语义等价——如果是，合并而非新增。

### 映射 6：记忆陈旧性 → 需要读取时检测

**问题本质**：记忆什么时候变得过时？

Walk-109 已经发现了这个问题：toxic memories trigger immune reaction, but stale memories look completely normal. 检测方法是用当前 skeleton 做显式对比——不一致的就是陈旧记忆。

Mem0 从工业角度提出了相似的方案：staleness detection 不应该依赖时间衰减（均匀老化），而应该依赖内容验证（与当前知识状态对比）。

**诊断**：Dreamer 已经在解决这个问题——walk-109 的"记忆陈旧性检测"规则 + subconscious 的 decay_rate 差异化。但还没有系统化的 staleness score。

## 洞察

**Insight 1**：Dreamer 的记忆架构虽然从未被设计为"正确的"，但它因为持续演进而意外覆盖了 6 个工业级记忆问题中的 4 个。缺失的两项（时间抽象 + 身份归一）恰好是可以通过增加 DORMANT 子步骤来解决的——不是架构级修改，是功能级补充。

**Insight 2**：六个未解问题构成一个相互依赖的六边形——解决任一个都需要先解决至少另一个。时间抽象需要先有身份归一（否则压缩会产生错误合并），身份归一需要先有应用级评估（否则无法判断合并是否正确）。这不是六个独立问题，是一个六体问题。

**Insight 3**：Dreamer 的解决方式不是"一步到位地解决问题"，而是"在散步中逐步发现症状→制定对策→验证效果"。这种演化式解法比一次性架构设计更适合开放域的记忆管理——因为记忆需求随系统演化而变化。
