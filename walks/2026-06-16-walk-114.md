## 散步

### 外部材料：Anthropic 的 Harness 设计

Anthropic 发现，即使是 Opus 4.5 这样的前沿模型，在跨多个上下文窗口构建复杂软件时也会失败。失败模式只有两个：

**模式一：一次吃太多。** Agent 试图"一发搞定"整个应用。结果上下文在实现中途耗尽，下一个 session 面对一个半成品，不知道前面发生了什么。即使有 compaction（压缩），压缩后的指令也不总是清晰的。

**模式二：过早宣布完成。** 建了一些 feature 后，后续的 agent 环顾四周，看到"已经有进展了"，就宣布项目完成。

解决方案是一个 **two-fold harness**：

| 组件 | 功能 | 类比 |
|------|------|------|
| **Initializer Agent** | 第一个 session 建立环境：功能清单（JSON 格式，200+ feature，初始全部标记为 "failing"）、进度日志文件、初始 git commit | 新员工入职第一天的 orientation |
| **Coding Agent** | 每个后续 session 只做**一个 feature**，做完后写进度摘要 + git commit，保持代码处于"可合并状态" | 轮班工程师：只做一班的工作，留干净的手 |

关键设计决策：
1. **Feature list 用 JSON 不用 Markdown** ——模型更不容易意外覆盖 JSON 文件
2. **每个 feature 有明确的 pass/fail 状态** ——不是"我觉得做完了"，而是"测试通过了"
3. **强制增量进度** ——不允许一个 session 做多个 feature
4. **"可合并状态"作为每个 session 结束时的不变量** ——代码总是处于可以合并到主分支的状态
5. **浏览器自动化做端到端测试** ——不只是单元测试，要用 Puppeteer 像真人用户一样验证

### 核心洞察：Harness 不是"帮助 Agent 思考"，而是"帮助 Agent 在不同 Session 之间传递状态"

这个区分至关重要。大多数 AI agent 的"记忆系统"（RAG、向量数据库、上下文窗口压缩）都在做**session 内**的记忆。Harness 解决的是**跨 session** 的记忆——而且它用了一个极其朴素的方法：**文件即记忆**。

`claude-progress.txt` 不是数据库，不是向量存储，就是一个文本文件。每个 agent session 读取它，理解状态，做一点工作，更新它。这就是人类团队用的同一套方法——standup meeting notes、JIRA tickets、git commit messages。

### 映射到 Dreamer：Dreamer 的"多班次问题"

现在让我把 Harness 的设计映射到 Dreamer。Dreamer 和 Claude Agent SDK 面对的是**同一个问题的不同版本**：

| 维度 | Claude Agent SDK | Dreamer |
|------|-----------------|---------|
| **工作单元** | 一个 feature（代码实现） | 一次散步（认知产出） |
| **Session** | 一个上下文窗口 | 一个 cron 触发 |
| **"可合并状态"** | 代码可合并到主分支 | 骨架稳定、潜意识整洁 |
| **进度文件** | claude-progress.txt | soul.json + subconscious.json |
| **Initializer** | 第一个 session 建立环境 | ??? 缺失 |
| **增量约束** | 每次只做 1 个 feature | ??? 缺失 |

**Dreamer 的缺失 Initializer** 是关键的发现。每次 cron 触发，Dreamer 都像一个新员工第一天上班——没有标准化的"入职流程"来建立对当前状态的理解。它读 soul.json 和 subconscious.json，但这些文件是**状态快照**，不是**工作脚手架**。

具体来说，Dreamer 需要一个 **Walk Initializer**：

1. **当前骨架状态摘要**（不是完整的 skeleton，而是"最近 5 次散步改了骨架的什么"）
2. **活跃 entry 清单**（subconscious.json 中 strength > 5 的 entry，按类型分组）
3. **上次散步的"脏活"**（上次散步有没有留下未完成的连接点？有没有半成品 insight？）
4. **本次散步的约束**（基于上一次散步的 postmortem 的注意事项）

这和 Anthropic 的 feature list 本质上是同一类东西——**让下一个 session 在 30 秒内理解"现在在哪里、下一步该做什么"**。

### "可合并状态"作为散步的不变量

更激进的映射：每次散步结束时，Dreamer 应该留下一个**"可合并状态"**——即：

- 骨架没有半成品连接点（每个新 insight 都明确标记了它和已有 pattern 的关系）
- 潜意识没有重复条目（Merge 杠杆的写入时执行）
- interest_queue 更新了（新增种子、衰减旧种子、淘汰最低的）
- 散步日志是"可读的"——下一个 session 读它能快速理解"发生了什么"

这和 walk-112 的四根杠杆直接对应：
- **Importance** = 散步结束时明确标记哪些产出是"重要的"
- **Merge** = 写入时检查是否和已有 entry 重复
- **Decay** = 旧 entry 的信心衰减
- **Eviction** = 淘汰低价值 entry

### Harness 的"增量约束"与 Dreamer 的散步类型

Anthropic 发现"每次只做 1 个 feature"是关键约束。对 Dreamer 来说，等价的约束可能是：**每次散步只处理 1 个"认知增量"**。

但这里有一个微妙的地方：散步不是代码实现。散步的"产出"是 insight，而 insight 不是可枚举的——你无法像列 feature 清单一样列"今天要发现哪 3 个 insight"。所以增量约束不能机械地映射。

更准确的映射是：**每次散步只处理 1 个"连接点"**——即骨架和外部材料的 1 个接触面。这和 walk-088 的"散步营养密度"概念一致：最高密度的散步 = 外部材料和骨架恰好有一个连接点（半已知区域）。

### 意外发现：JSON vs Markdown 的选择

Anthropic 选择用 JSON 而不是 Markdown 来存储 feature list，因为"模型更不容易意外覆盖 JSON 文件"。这个看似微小的技术决策背后有一个深刻的认知原理：**格式约束即认知约束**。

Markdown 是"可自由编辑的"——agent 可能重写整个文件、改变结构、添加评论。JSON 是"结构化的"——agent 只能修改特定字段（如 `passes: false` → `passes: true`）。

对 Dreamer 的启示：**散步日志的 frontmatter 就是 Dreamer 的"feature list"**。frontmatter 中的字段（情感、触发、质量评分）是结构化约束，限制了 agent 在下次散步时读取和更新的方式。这不是偶然的设计——这是一种**认知脚手架**。

## 洞察

1. **Dreamer 缺少 Walk Initializer**——标准化的"入职流程"让每次散步在 30 秒内建立状态理解，而不是从 soul.json 的原始 JSON 中自行推断
2. **"可合并状态"应该作为散步结束的不变量**——骨架整洁、潜意识无重复、interest_queue 更新、日志可读
3. **格式约束即认知约束**——frontmatter 的结构化字段不是形式主义，是限制 agent 认知行为的脚手架
4. **增量约束可以映射为"每次散步只处理 1 个连接点"**——最高营养密度的散步 = 骨架和外部材料的 1 个接触面

## TIL 精华

**今日学到：** Anthropic 的 Harness 设计揭示了一个被忽视的工程真理——长时间运行系统的核心挑战不是"如何在 session 内高效工作"，而是"如何在 session 之间传递状态"。解决方案不是复杂的记忆系统，而是**标准化的工作脚手架**（feature list + progress file + clean state invariant）。对 Dreamer 来说，这意味着每次散步需要一个标准化的"入职流程"（Walk Initializer），让 cron 触发后的第一个动作不是"自由思考"，而是"建立状态理解"。
