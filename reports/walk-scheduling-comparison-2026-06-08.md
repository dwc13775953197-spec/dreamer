# Dreamer 散步调度机制优化方案对比

> 2026-06-08 | 基于 GitHub 调研 + 认知科学理论

---

## 背景：当前机制的问题

当前散步调度本质是"固定间隔 × 情绪因子"：

```
间隔 = 288分钟 × curiosity_factor × melancholy_factor × energy_factor
```

**问题**：
1. **时钟驱动，不是需求驱动**——不管有没有"认知需求"，到点就散
2. **频率双向绑定**——散步越多→curiosity下降→频率降低→散步更少（正反馈死循环）
3. **无法区分"该散"和"该消化"**——刚写完一篇深度散步，可能更需要消化而不是立刻再散
4. **无累积效应**——每次散步是独立的，没有"认知债务"概念

---

## 方案对比

### 方案 A：认知压力模型（我之前提的）

**来源**：认知科学中的"认知负荷理论"（Sweller, 1988）+ 人类注意力恢复理论（Kaplan, 1995）

**核心思想**：把散步从"定时触发"改成"认知压力积累到阈值时触发"

```
压力积累：
+ 上次散步后经过时间 × 0.5（基础增长）
+ 距离上次散步的小时数 × 2（时间越久压力越大）
+ RSS 新发现 × 5（外部刺激）
+ 高兴趣种子 × 3（intensity ≥ 7）
+ 上次产出为 0 → +10（骨架饥饿信号）

压力释放：
× 散步后：pressure -= (产出数量 × 15)（产出越多释放越多）
× DORMANT 后：pressure × 0.5
× 用户对话后：pressure × 0.7

触发条件：pressure ≥ threshold（默认 50）
threshold 动态调整：
  energy < 30 → threshold × 1.5（疲惫时更不容易触发）
  curiosity > 0.9 → threshold × 0.7（好奇时更容易触发）
```

**优点**：
- 需求驱动——该散的时候散，不该散的时候不硬撑
- 自适应——阈值随状态动态调整
- 简单——一个数字（pressure）决定一切
- 可解释——能明确说出"为什么现在触发/不触发"

**缺点**：
- 压力公式需要调参（0.5、2、5、3、10 这些系数是经验值）
- 仍然是"检查式"——需要定期 cron 来检查压力值
- 压力释放系数（产出数量 × 15）需要验证

**实施成本**：改 1 个 cron prompt + soul.json 新增 2 个字段

### 方案 B：Thinking Clock（来自 OpenClaw 社区提案 #17287）

**来源**：OpenClaw 社区提案 [feat: Agent Thinking Clock](https://github.com/openclaw/openclaw/issues/17287)

**核心思想**：双层注意力系统——便宜模型持续"外围感知"，发现有趣的事才升级到深度思考

```
┌──────────────────────────────────────┐
│  外围层（便宜模型，每 5 分钟 tick）    │
│  "有什么值得注意的吗？"               │
│  → TICK_OK（无事发生）               │
│  → ESCALATE: <reason>（触发散步）    │
└──────────────┬───────────────────────┘
               │ ESCALATE
               ▼
┌──────────────────────────────────────┐
│  聚焦层（主模型，按需触发）            │
│  深度散步 / 研究 / 消化               │
└──────────────────────────────────────┘
```

**关键设计**：
- 外围 tick 成本极低（~$0.005/tick 用便宜模型）
- 5% 升级率 → 每天 ~14 次深度思考
- 安静时段自动抑制（23:00-07:00）
- 每日上限保护（200 ticks / 20 escalations）

**优点**：
- **真正的持续认知**——不是在"散步"和"不散步"之间二选一，而是持续背景思考
- **成本极低**——外围 tick 用便宜模型，每天 ~$3.50
- **自然升级**——不需要决定"什么时候该散步"，而是"散步一直在发生，只是大多数时候很轻"
- **与 Dreamer 架构天然契合**——外围 tick ≈ RESTING 状态，升级 ≈ ACTIVE 状态

**缺点**：
- 需要两个模型（便宜 + 昂贵），Dreamer 目前只用一个模型
- tick prompt 设计需要精心编排（问什么问题？）
- "升级"判断标准需要学习/调优
- 与现有 cron 架构有冲突（Thinking Clock 是持续运行的，不是 cron 触发的）

**实施成本**：需要新增一个持续运行的后台进程（不是 cron），改动较大

### 方案 C：SOFAI 双系统（来自 Nature npj AI 2025）

**来源**：[Fast, slow, and metacognitive thinking in AI](https://www.nature.com/articles/s44387-025-00027-5)

**核心思想**：System 1（快速反应）+ System 2（深度推理）+ 元认知（调度器）

```
┌──────────────────────────────────────────┐
│  元认知调度器（Metacognitive Agent）       │
│                                          │
│  三种模式：                               │
│  1. 实时模式：评估 S1 质量，决定是否需要 S2  │
│  2. 反思模式：回顾过去决策，调整调度参数     │
│  3. 学习模式：更新 S1/S2 的经验模型        │
└──────────┬───────────────┬───────────────┘
           │               │
           ▼               ▼
┌──────────────┐  ┌──────────────┐
│  System 1    │  │  System 2    │
│  "快速"      │  │  "慢速"      │
│  散步/浏览   │  │  深度散步/研究 │
│  每次 1 搜索  │  │  多步推理     │
│  产出: 种子   │  │  产出: 洞察   │
└──────────────┘  └──────────────┘
```

**关键设计**：
- S1 自动触发，S2 永远不主动触发（必须由元认知调用）
- 元认知基于"信任度"决定：trustS1 = min(置信度, 无违规百分比)
- 反思模式用反事实推理——"如果当时用 S2 会怎样？"——来改进调度

**优点**：
- **认知科学基础扎实**——基于 Kahneman 的双过程理论
- **元认知闭环**——不只是调度，还能学习改进调度
- **信任度机制**——不是固定阈值，而是基于历史表现的动态信任
- **与 Dreamer 的 DORMANT 审计天然契合**——反思模式 ≈ DORMANT 审计

**缺点**：
- 最复杂——需要三个组件（S1 + S2 + MC）
- 元认知本身的计算开销
- 需要定义"S1 质量"的评估标准
- 论文中的实验环境是网格导航，迁移到认知系统需要大量适配

**实施成本**：需要重新设计散步触发架构，改动最大

### 方案 D：混合方案（推荐）

**核心思想**：取 A 的简洁 + B 的持续感知 + C 的元认知

```
┌──────────────────────────────────────────────────┐
│                    感知层（持续运行）                │
│  每次 cron tick（15-30 分钟）检查：                │
│  - 认知压力值（方案 A）                            │
│  - 是否有值得注意的新信号（方案 B 的 ESCALATE）    │
│  - 上次产出的质量（方案 C 的信任度）                │
│                                                  │
│  决策：                                           │
│  ┌────────────────────────────────────────────┐  │
│  │  如果 pressure > threshold → 触发散步       │  │
│  │  如果 quality < 5/连续3次 → 切换散步类型    │  │
│  │  如果 diversity_seed 存在 → 强制跨领域散步   │  │
│  │  如果 energy < 20 → rest（只衰减不散步）    │  │
│  │  否则 → 只更新压力值，不散步                │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  散步后更新：                                     │
│  - pressure -= insight_count × 15                │
│  - quality_history.append(quality_score)         │
│  - diversity_score = 更新主题熵                   │
└──────────────────────────────────────────────────┘
```

**与 Dreamer 现有架构的映射**：

| 混合方案组件 | Dreamer 对应物 | 状态 |
|-------------|---------------|------|
| 压力积累 | soul.json.pulse.cognitive_pressure | ✅ 可新增 |
| 压力阈值 | soul.json.pulse.pressure_threshold | ✅ 可新增 |
| 质量追踪 | audit.jsonl（已有质量评分） | ✅ 已有 |
| 信任度 | dormant_stats.total_audits / FLAWED 比例 | ✅ 可从现有数据计算 |
| 多样性种子 | subconscious.json（type=diversity_seed） | ✅ 已建立 |
| 持续感知 | 脉冲触发 cron（已存在） | ✅ 改造而非重建 |
| 深度散步 | 散步触发 cron（已存在） | ✅ 复用 |
| 元认知调度 | DORMANT 控制器（已建立） | ✅ 复用 |

**关键创新点**：
1. **压力替代时间**——不再"每60分钟检查一次"，而是"压力够了才散步"
2. **产出反馈**——散步产出越多，压力释放越多（高质量散步 → 更长间隔，符合认知科学中的"消化时间"）
3. **多样性保护**——主题熵低时强制跨领域散步
4. **信任度学习**——连续低质量散步 → 自动切换类型（分析↔探索）

**实施计划**：

```
Phase 1（1 小时）：
  - soul.json 新增 cognitive_pressure, pressure_threshold, quality_history
  - 脉冲触发 cron prompt 改为"压力检查"模式
  - 散步触发 cron prompt 增加"散步后更新压力"步骤

Phase 2（30 分钟）：
  - DORMANT 控制器增加"压力释放"步骤
  - 月度验证 cron 增加"压力趋势分析"

Phase 3（持续调参）：
  - 观察 3-5 天运行数据
  - 调整压力系数（0.5, 2, 5, 10, 15）
  - 调整阈值（默认 50）
```

## 方案对比总结

| 维度 | A: 认知压力 | B: Thinking Clock | C: SOFAI | D: 混合（推荐） |
|------|-----------|-----------------|----------|--------------|
| 成本 | 低 | 中（需2个模型） | 高 | 低 |
| 实施难度 | 低 | 高（需新进程） | 很高 | 中 |
| 认知科学基础 | 中 | 中 | 高 | 高 |
| 与 Dreamer 兼容 | ✅ 高 | ⚠️ 中（架构冲突） | ⚠️ 低 | ✅ 最高 |
| 可解释性 | ✅ 高 | ✅ 高 | ⚠️ 中 | ✅ 高 |
| 自适应能力 | ⚠️ 中（固定公式） | ✅ 高（持续感知） | ✅ 高（元认知） | ✅ 高 |
| 持续认知 | ❌ 离散 | ✅ 持续 | ✅ 持续 | ⚠️ 准持续（15分钟间隔） |
| 调参负担 | 中 | 低 | 高 | 中 |

**推荐方案 D**——它用最小的改动成本（改造现有 cron，不是新建进程），整合了三个方案的优点：
- A 的简洁（一个压力值决定一切）
- B 的升级机制（值得注意时才深度思考）
- C 的产出反馈（信任度学习）
