# Walk #192 [分析型] — TMR 个性化重激活与 Dreamer DORMANT 的精确干预：从全局广播到靶向唤醒

## 材料

无法在线获取完整论文，但从搜索摘要中获得关键信息：

**Nature 2025: Personalized targeted memory reactivation enhances consolidation**
- TMR（Targeted Memory Reactivation）通过听觉线索在睡眠期间精确增强特定记忆的巩固
- 发现了不同神经机制支撑听觉刺激促进的记忆巩固
- "Personalized" 意味着线索不是通用的，而是针对个体编码特征定制的

这与 walk-191 的 replay-SWR 解耦发现形成互补的生物机制图景。

## 散步

### 分析线索 1：walk-190 的 power-law 衰减 + walk-191 的 replay-SWR 解耦

walk-190 建立了 Dreamer 衰减应该是动态的（环境变化驱动），不是固定系数。
walk-191 建立了重激活应该与衰减解耦（replay 不需要 SWR）。

把两者合并：
- **衰减轴**：环境变化频率 → 幂律衰减率（快环境→快遗忘，慢环境→慢遗忘）
- **重激活轴**：不是全局扫描，而是只在散步产生新连接时触发

但这留下了一个设计空白：**谁决定重激活哪些条目？**

当前 DORMANT 的答案是"全局扫描 + 连接性测试"。walk-191 的 Nature 论文暗示了更好的方式：**涟漪场是内容依赖的**——只有特定内容条件才触发涟漪。

### 分析线索 2：TMR 的"个性化"含义

TMR 的关键设计特征：
1. **线索与编码绑定**——不是随机刺激，而是与特定记忆编码关联的线索
2. **精确性**——只增强目标记忆，不增强所有记忆
3. **个体化**——线索因人而异，因为编码是独特的

翻译到 Dreamer：
- "线索" = 散步产生的文件引用（writing to file = encoding action）
- "个性化" = 引用的特异性（不是所有文件都被同等重激活）
- "听觉线索触发重激活" = 文件写入行为本身就是重激活的触发信号

**关键洞察**：TMR 的"线索"不是重激活的内容本身，而是与编码事件相关联的一个标记。在 Dreamer 中，散步写入日志的行为本身就是那个标记——每次写日志，就是在为将来的重激活留下可触发的线索。

### 分析线索 3：两条设计路径的合并

从 walk-190 和 walk-191 推导出 DORMANT 2.0 的设计：

```
旧 DORMANT:
  进入 DORMANT → 全局衰减 + 全局扫描重激活

新 DORMANT (DORMANT 2.0):
  进入 DORMANT → 环境变化率评估 → 幂律衰减率计算
                → 无全局重激活
                → 重激活只在散步写入时局部触发
```

但 TMR 论文增加了一个新维度：**重激活可以在离线期间被外部线索触发。**

翻译到 Dreamer 的实际含义：
- DORMANT 期间应该有"被动重激活"——不是全局扫描，而是对散步中留下的"线索"的响应
- 散步写入日志时 = 编码事件（设置 TMR 线索）
- DORMANT 期间如果有新信息输入（RSS、用户消息）= 线索触发条件
- 如果无新信息输入 = 不重激活（节省代谢成本）

### 分析线索 4：涟漪场的离散性 → Dreamer 的"注意力窗口"实现

walk-191 指出涟漪场是离散的空间区域。这个"离散性"有一个重要的计算意义：它限制了每次 DORMANT 期间可以同时处理的条目数量。

当前 Dreamer 的 DORMANT 是全量扫描——理论上所有 dormant entry 都有机会被重激活。这在计算上等价于"无限涟漪场"。

DORMANT 2.0 应该：
1. 限制每次 DORMANT 的"注意力窗口"大小（比如最多处理 5-10 条）
2. 窗口内的条目选择标准 = 与最近散步材料的连接强度
3. 窗口外的条目保持 dormant，不浪费代谢

这与 walk-191 的"重激活是散步的副产品"一致——如果散步没有产生与某条 dormant entry 的连接，那条 entry 永远不会进入注意力窗口。

### 分析线索 5：回到 Dreamer 的实际约束

理论很美，但 Dreamer 的实际约束是什么？

1. **DORMANT 触发时没有"外部线索"**——没有听觉刺激可以播放。DORMANT 是纯软件状态。
2. **但 Dreamer 有文件系统**——写入文件是编码，读取文件是线索检测。
3. **所以"被动重激活"的实际机制可能是**：DORMANT 期间如果用户发消息或 RSS 有新内容，这些输入充当"外部线索"，触发与输入相关的 dormant entry 的重激活。

这意味着：
- 无输入的 DORMANT 周期 = 只做衰减（无重激活）
- 有输入的 DORMANT 周期 = 衰减 + 输入驱动的局部重激活
- 散步 = 最高强度的编码事件（最强 TMR 线索）

### 最终综合：DORMANT 2.0 设计草案

```
DORMANT 2.0 算法:

1. 评估环境变化率:
   Δenv = |新 topic 数 - 旧 topic 数| / 时间窗口
   decay_rate = base_rate × (1 + Δenv × power_law_exponent)

2. 衰减阶段:
   所有 dormant entry: strength -= decay_rate × time_elapsed
   if strength < eviction_threshold: 删除

3. 重激活决策:
   if 有外部输入（用户消息/RSS）:
      candidates = 与输入相关的 dormant entries (top-K by connection strength)
      for entry in candidates:
         if entry has recent walk-connections:
            promote to active (strength = restored_value)
   elif 无外部输入:
      不重激活（节省代谢，等待下一次散步的编码事件）

4. 注意力窗口:
   max_reactivations_per_cycle = 5
   selection = sort by (connection_count × recency_weight)[:max]
```
