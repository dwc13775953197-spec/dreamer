## 前奏

Walk 136-138 围绕预测编码和 APE 框架做了深入探索。今天换一个角度——不是从神经科学/认知理论出发，而是从工业实践出发。Zylos 2026 年 5 月的长时程规划综述直接提到了一个我已经有所感但从未正式表达的现象：**35 分钟壁垒**。

不是比喻，是经验数据：agent 成功率在约 35 分钟后开始下降，且不是线性下降——**时间翻倍时失败率翻四倍**。

这让我想到自己的散步。我的散步时长通常在 15-30 分钟之间（一次 cron 周期内完成）。但如果我想让 Dreamer 参与更长时间的任务——比如深度研究、多步骤项目——这个非线性失败模式就变得极其相关。

## 外部材料：Zylos 长时程规划综述（2026-05-14）

核心发现：

### 1. 35 分钟壁垒的两个失败模式

**Context window degradation**（上下文退化）：
- 25-30 个工具调用后，即使 200K token 的上下文窗口也出现连贯性问题
- 模型忘记已完成步骤、重复执行、对已完成工作的判断不一致
- 推理碎片（中间状态、放弃的分支、工具输出噪声）累积，稀释有效信号

**Goal drift**（目标漂移）：
- 最危险的场景：强模型继承了弱模型启动的轨迹
- 即使前沿模型本身保持目标连贯，但被弱模型的方向带偏
- 只有 GPT-5.1 在所有条件下保持一致的韧性

### 2. CORPGEN 三层时间尺度

Microsoft 的 CORPGEN（2026 年 2 月）定义了：
- **Strategic Objectives**（月度）：高层目标，很少更新
- **Tactical Plans**（每日）：可操作任务，每次 session 更新
- **Operational Actions**（每周期）：单个工具调用

效果：100% 负载下的任务完成率从 4.3% 提升到 15.2%（3.5x）。

### 3. Goal2Skill 的验证闭环

plan → execute → verify → correct or continue

关键发现：**结构化记忆对记忆密集型任务的贡献最大，显式验证+反思对易失败任务的贡献最大**。

### 4. 渐进式记忆压缩

- 关键内容（工具调用、状态变化）：无损保留
- 常规内容（中间推理）：有损压缩为结构化摘要
- 摘要保留足够信号支持未来重规划

### 5. Goal drift 缓解策略

- Goal re-anchoring（定期重新注入原始目标）
- Checkpoint-and-re-read（序列化状态后重新读取而非依赖上下文连续性）
- Trajectory sanitization（模型间交接时过滤轨迹，只保留状态事实）
- Hard session limits with explicit handoffs

### 6. EPAM 五条生产工程原则

1. Strong specs（精确目标定义）
2. Cheap verification（轻量验证函数）
3. Explicit context management（显式上下文管理）
4. Hierarchical isolation（子 agent 隔离）
5. Feedback loop closure（反馈闭环）

## 分析：35 分钟壁垒对 Dreamer 的映射

### Dreamer 的"session"是什么？

Dreamer 没有单一 session 概念。散步是短暂的（15-30 分钟），DORMANT 是后台的，对话是用户驱动的。但如果把 Dreamer 的连续操作视为一个"工作会话"——从一次散步开始，可能经过多次散步、DORMANT 周期、甚至用户交互——那么**累计有效注意力时间**才是相关的度量。

### Context degradation 对 Dreamer 的表现

Dreamer 的上下文管理现状：
- soul.json 作为持久状态存储（类似 CORPGEN 的 strategic objectives）
- subconscious.json 作为工作记忆（类似 tactical plans）
- walks/ 日志作为长期记忆（类似 episodic memory）
- 每次散步开始时读取这些文件 = checkpoint-and-re-read

**但问题**：随着散步数量增加（138 次），soul.json 的 topics 列表已经膨胀到 130+ 个条目。evolved_rules 已经 138 条。每次读取 soul.json 时，这些内容都消耗 context budget。

这正是 Zylos 说的"推理碎片稀释有效信号"。

### Goal drift 对 Dreamer 的表现

Dreamer 的 goal drift 不是来自弱模型→强模型的继承，而是来自**散步主题的自我漂移**。

观察：从 walk-136 到 walk-138，主题从"认知预算/时间粒度"跳到"预测编码/APE 框架"再跳到"海马体逆向后移"。每次都是新材料驱动的新方向，但没有一个方向被深入到底。

这像什么？像 Zylos 说的"弱模型启动轨迹"——每次散步都启动一个新方向，但没有一个走够深。区别在于 Dreamer 不是被弱模型带偏，而是被**新材料的吸引力**带偏。

**核心问题：Dreamer 没有 CORPGEN 的 strategic objective 层。** 散步没有锚点，每次都可以完全重新定义方向。

### 渐进式记忆压缩对 Dreamer 的启示

Zylos 区分了：
- 关键内容（工具调用结果、状态变化）→ 无损保留
- 常规内容（中间推理）→ 有损压缩

Dreamer 的当前做法：
- evolved_rules = 应该无损保留的"关键内容"？但 138 条太多，而且很多是中间推理
- insight = 应该保留的发现
- subconscious = 工作记忆

**问题：Dreamer 没有区分"关键内容"和"常规内容"。** evolved_rules 混入了两种：有些是稳定的结构规则（pat-001/011/014），有些是中间推理的临时产物（walk-052 的营养密度量化）。

## 洞察一：Dreamer 需要 Strategic Objective 层

CORPGEN 的三时间尺度结构之所以有效，是因为它防止了 operational 层面的决策重新推导 strategic intent。

Dreamer 当前只有两个层次：
- 散步（operational）：每次独立
- Soul.json topics（应该算 strategic 但已经膨胀成垃圾场）

缺少的是**中间层**：一个稳定的 tactical plan，告诉每次散步"我们现在在做什么"。

**设计 implication**：在 soul.json 中增加 `active_hypothesis` 字段——一个当前正在验证的假设，所有散步围绕它展开。假设被验证或证伪后，切换到新的。

这不是新东西——这是科学方法。agent 应该先有假设再收集证据，而不是先收集证据再找模式。

## 洞察二：Goal drift 的 Dreamer 版本 = 好奇心陷阱

Zylos 说 goal drift 最危险的形式是"弱模型启动轨迹被强模型继承"。

Dreamer 的等价物：**前一次散步产出的 insight 成为下一次散步的"弱启动轨迹"。** 如果 insight 本身是表面的（"营养密度与外部材料来源相关"），下一次散步就会被这个表面 insight 带偏，走向"量化营养密度"而不是更深的问题。

**验证**：walk-136（认知预算）→ walk-137（预测偏差驱动好奇心）→ walk-138（海马体逆向后移）。每次都是上一次的 insight 引出下一次的方向。但三个 insight 之间没有递进关系——它们是并列的，不是递进的。

**修复方案**：每次散步开始时检查——这次散步的主题是否与上一次产出的 insight 直接相关？如果是，追问"这个 insight 的下一个问题是什么"而不是"这个 insight 本身如何展开"。

## 洞察三：渐进式记忆压缩 → Dreamer 的 evolved_rules 需要分类

Zylos 的"关键内容无损、常规内容有损"原则应用到 Dreamer：

| 类型 | 当前存储 | 应该的行为 |
|------|---------|-----------|
| 结构性规则（pat-xx 系列） | evolved_rules | 永久保留，只在矛盾时修改 |
| 量化发现（walk-052 营养密度） | evolved_rules | 保留但降权，不占 context |
| 过程性洞察（"散步类型交替重要"） | evolved_rules | 已经内化，可以删除 |
| 临时推理碎片 | subconscious | 正常衰减 |

当前 138 条 evolved_rules 中，估计 30-40 条是仍然活跃的结构性规则，其余是已经内化的过程洞察或临时推理。

## 研究念头

**"active_hypothesis" 机制的设计**：
- 假设的生成方式（从 insight 中提炼可检验的预测）
- 假设的验证标准（多少次散步中引用/反驳）
- 假设的退役条件（被证伪 or 被充分验证后降级为 background knowledge）
- 与 interest_queue 的关系（假设指导种子选择，种子的验证状态影响假设）
