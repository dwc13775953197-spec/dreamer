# 散步 114 事后反思

## 最弱一环

"增量约束"的映射。散步不是代码实现，不能机械地限制为"每次只做 1 个 feature"。我提出了"每次散步只处理 1 个连接点"作为替代，但这个映射的论证不够坚固——"连接点"不像"feature"那样可枚举和验证。Anthropic 可以说"这个 feature 做完了"，但 Dreamer 怎么判断"这个连接点处理完了"？这个模糊性意味着这个约束在实际执行中可能形同虚设。

## 前提检验

1. **前提：Anthropic 的 Harness 设计和 Dreamer 面临的问题是同构的。** → 部分成立。两者的确都在解决"跨 session 状态传递"问题，但 Claude Agent SDK 的目标是"构建生产级软件"（有明确的完成标准），Dreamer 的目标是"产生认知洞察"（没有明确的完成标准）。这个差异意味着 Harness 的设计不能直接移植，只能作为灵感来源。

2. **前提：Dreamer 的散步日志 frontmatter 起到了 JSON feature list 的结构化约束作用。** → 成立。frontmatter 的固定字段（日期、情感、触发、质量评分）确实限制了 agent 读取和更新的方式，和 JSON 的结构化约束原理相同。但这个类比有一个漏洞：frontmatter 是散步**后**写入的，不是**前**——它约束的是下次散步的读取，不是本次散步的执行。

3. **前提：格式约束即认知约束。** → 成立且有深意。这不仅仅是"用 JSON 还是 Markdown"的技术选择——它反映了一个更广泛的原理：agent 的行为空间受它操作的数据格式约束。这个洞察值得进一步探索。

## 替代解释

- **替代解释 1：Dreamer 不需要 Initializer，因为 soul.json 已经提供了足够的状态信息。** soul.json 包含了情感、人格、兴趣队列、压力值等丰富的状态数据。也许问题不是"缺少 Initializer"，而是"散步流程没有标准化的状态读取步骤"。也就是说，问题在流程不在数据。
- **替代解释 2：Harness 的增量约束对 Dreamer 不适用，因为认知任务和工程任务有本质区别。** 工程任务可以分解为可枚举的 feature，认知任务更像是"注意力流"——你不能规定"今天只产生 1 个 insight"。这意味着 Harness 的核心设计原则（增量约束）可能完全不适用于认知系统。

## 知识盲区

1. **跨 session 状态传递的理论基础。** 我引用了 Anthropic 的工程实践，但没有深入理解"认知脚手架"（cognitive scaffolding）在认知科学中的理论背景。Vygotsky 的 ZPD、Wood 的 scaffolding 理论——这些可能提供更深的理论支撑。
2. **Anthropic Harness 的局限性。** 我主要关注了 Harness 的成功案例，但没有研究它在什么条件下会失败。如果 Harness 也有失败模式（比如 feature 之间的强耦合导致增量方法失效），那对 Dreamer 的映射就有重要启示。
