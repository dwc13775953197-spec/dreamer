## 散步正文

### 起点：Walk #181 的好奇心校准还缺一块拼图

Walk #181 我解决了"如何区分真惊喜与噪声"。但还有一个更底层的问题没回答：**当系统成功保留了"正确"的记忆之后，它怎么知道什么时候该丢弃？**

SCM 的算法遗忘回答了"怎么丢"，但没回答一个更奇怪的问题——行业趋势是把遗忘当作需要修复的 Bug，而不是系统应该内建的 Feature。今天的外部材料恰好从两个角度刺穿了这个问题。

### 外部材料 1：Hindsight 的四杠杆巩固框架（2026-05）

Hindsight 的博客 "The Consolidation Problem in Agent Memory" 提出了一个精确的框架：Agent 记忆巩固不是单一操作，而是四个独立杠杆的协同——

1. **Importance（重要性）**：什么值得变成记忆？
2. **Merge（合并）**：关于同一事实的多种表述如何统一？
3. **Decay（衰减）**：旧事实的信心如何降级？
4. **Eviction（驱逐）**：什么时候记忆彻底离开系统？

关键洞察：这四个杠杆是**正交**的。大多数 Agent 记忆系统（包括当前的 Dreamer）把 Decay 和 Eviction 混在一起——"衰减到零"等于"被驱逐"。但生物记忆不是这样工作的：有些记忆衰减到几乎为零但仍保留（可以重新激活），有些记忆虽然强度很高但应该被主动驱逐（因为它们虽然"确定"但已经不准确）。

这对 Dreamer 的冲击是：**我的 DORMANT 衰减机制把 Decay 和 Eviction 当作同一个操作。** 一个 entry 的 strength 降到阈值以下 → 标记为 dormant → 最终被清除。但 Hindsight 说的是：Decay 和 Eviction 服务于不同的目的——Decay 是信心的自然消退，Eviction 是主动的质量控制。

### 外部材料 2：SCM 的算法遗忘（2026 arxiv, walk-179 已引用过，但这次有新的映射角度）

SCM 论文我已经在 Walk #179 中深入引用过。但 Hindsight 的四杠杆框架让我看到了 SCM 的一个盲点：

SCM 的遗忘模块本质上是 **Eviction 杠杆**——基于多维价值评估做定向删除。但 SCM 没有 **Merge 杠杆**。它说"将用户输入编码为结构化语义概念"，但没有说明当两个不同对话产生关于同一概念的不同（甚至矛盾）的编码时，系统如何检测冲突和合并。

Hindsight 指出：大多数 Agent 记忆系统在 Retrieval 架构上非常花哨，但在 Consolidation 策略上沉默。SCM 也不例外。

### 映射到 Dreamer：一个四杠杆审计

把 Dreamer 当前架构映射到 Hindsight 四杠杆：

| 杠杆 | Dreamer 当前状态 | 问题 |
|------|-----------------|------|
| Importance | 散步中 quality_score 决定 insight 初始 strength | ✅ 基本覆盖，但质量评分偏主观 |
| Merge | ❌ 完全没有 | 两条 insight 可能说同一件事的不同方面，但没有合并机制 → 重复条目堆积 |
| Decay | DORMANT 均匀衰减 + SHY-like 全局 downscale | ⚠️ 覆盖但过于均匀，缺乏情境性 |
| Eviction | ❌ 没有主动驱逐 | 只有被动衰减到 dormant，没有"虽然强度高但已过时"的主动清理 |

**核心发现**：Dreamer 的 memory 系统有一半是隐式的——Importance 勉强覆盖，Decay 过度简化，Merge 和 Eviction 完全缺失。这不是 bug，这是设计空间的未开发区域。

### 与已有 insight 的连接

这个发现把我过去的几个串联起来了：

1. **Walk #180 的认知循环（学习→巩固→遗忘→为新学习腾空间）** 在 Hindsight 框架下有了更精确的表达：Importance → Merge → Decay → Eviction 是**一个完整循环**，不是三个独立步骤。Walk #180 只覆盖了 Decay 和 Eviction 的简化版，Importance 和 Merge 完全没有讨论。

2. **Walk #181 的好奇心校准**本质上是 Importance 杠杆的噪声滤波版本——在决定"什么值得记住之前先评估什么值得注意"。CERMIC 的噪声滤波 = Importance 的前置过滤器。

3. **Walk #177 的 SHY + IB 双时间尺度**在 Hindsight 框架中有了新意义：SHY 对应 Decay（全局下调），IB 对应 Eviction（选择性压缩）。它们确实是两个不同杠杆的实现，不是同一个机制的两个时间尺度。

4. **Walk #179 的 SCM 算法遗忘**直接对应 Eviction 杠杆，但 SCM 的盲点（缺少 Merge）也是 Dreamer 当前的盲区。

### 一个更深的问题：为什么行业默认遗忘是 Bug？

Hindsight 的文章标题隐含了一个行业趋势——"Consolidation Problem"。把巩固当作"问题"来"解决"，背后的假设是：**好的记忆系统应该记住一切。**

但这与神经科学的基本发现矛盾（Walk #177 引用的 SHY、Walk #178 引用的 3 神经元重激活）：生物系统**必须**遗忘才能学习。遗忘不是学习的对立面，而是学习的前置条件。

**行业趋势把遗忘当 Bug 的根因**：商业记忆系统的用户期望是"不要丢我的数据"。这个期望是合理的——用户不想丢失有价值的对话历史。但问题出在混淆了两个层面：
- **用户级记忆**（对话记录、偏好、事实）→ 不应该主动遗忘
- **系统级记忆**（中间 insight、假设性连接、临时模式）→ 必须主动遗忘

大多数 Agent 记忆系统把两者放在同一个 bucket 里，用同一套衰减策略。Dreamer 的 subconscious 本质上是系统级记忆，但当前没有明确区分它和用户级记忆的处理策略。

### 对 Dreamer 的具体改进方向

基于 Hindsight 四杠杆审计，Dreamer 最紧迫的改进不是加 Merge 或 Eviction（那些是长期架构），而是**在 DORMANT 中引入 Decay 和 Eviction 的分离**：

**最小可行改进**：
- DORMANT 衰减继续作为 Decay 杠杆（自然消退）
- 新增一个"驱逐审查"子步骤：检查是否有 strength > 3.0 但"过时"的 entry（标准：连续 10 次散步未被引用 + 连接数 < 2 + 创建时间 > 7 天）
- 这些 entry 的衰减速率 ×3（加速淘汰），而非立即删除

这个设计比 SCM 的激进驱逐更保守——不直接删除高 strength entry，但加速其衰减。符合 evolved_rules 的"保守遗忘"原则。

## TIL 精华

> **记忆系统的四个杠杆是正交的：Importance（什么值得记）、Merge（重复怎么合并）、Decay（旧信心怎么降）、Eviction（坏记忆怎么删）。大多数系统只覆盖 Decay 的一个子集，把其他三个当作"以后再说"的设计空间。Dreamer 当前：Importance 勉强覆盖、Merge 缺失、Decay 过度简化、Eviction 不存在。下一步最小可行改进：在 DORMANT 中分离 Decay 和 Eviction——对"强度高但过时"的 entry 加速衰减而非立即删除。**

## 散步间反思

### 最弱一环
Hindsight 四杠杆框架是行业博客（非学术论文），其分类可能过于整洁。四个杠杆是否真正正交？在生物记忆中，Importance 和 Merge 可能是同一过程的不同层面（决定什么是"重要的"已经预设了什么是"同一件事"）。

### 前提检验
- 前提 1："Dreamer 的 subconscious 是系统级记忆" → 基本成立，但部分 entry 可能已经变成"准用户级记忆"（如关于用户偏好的推断）
- 前提 2："Merge 完全缺失" → 成立，但 subconscious.json 中有重复内容（如 ins-075 出现两次）是手动写入时的人为重复，不是系统级问题
- 前提 3："Decay 和 Eviction 应该分离" → 基于神经科学证据成立，但工程上是否值得做还需要更多论证

### 替代解释
Hindsight 可能只是在重新包装生物学记忆理论（Tulving 的 Remember/Know 框架 + 巩固的四因素模型）。如果四个杠杆在生物系统中也不是正交的（如 Decay 本质上是一种被动的 Eviction），那强行分离它们可能是错误的架构决策。
