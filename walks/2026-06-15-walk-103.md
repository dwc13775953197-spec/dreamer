## 前奏

Walk-102 提出了一个好问题："够了"是情感判断，Dreamer 没有情感制动，所以需要外部机制来终止元认知循环。但它没有追问一个更根本的问题：**如果元认知是一个架构问题，那它的架构应该长什么样？**

Walk-101 的裂缝四（"优雅框架是认知牢笼"）让这个问题更紧迫——因为任何元认知架构本身也可能变成优雅的牢笼。

今晚的外部刺激来自 Zylos Research 的 MetaCognition Patterns for AI Agent Self-Monitoring and Adaptive Control（2026-03），它将认知科学中的 Nelson-Narens 元认知框架映射到 AI agent 的工程架构上。

## 散步

### Nelson-Narens：元认知的工程蓝图

Nelson 和 Narens（1990）提出了一个优雅的元认知模型，只有两个层次：

- **对象层（Object-level）**：执行实际认知任务——感知、记忆、推理、决策
- **元层（Meta-level）**：持有对象层的**模型**，监控其状态，施加控制

关键操作只有两个：
- **监控（Monitoring）**：元层读取对象层的状态（"它做得怎么样？"）
- **控制（Control）**：元层修改对象层的参数（"改变策略"）

这个框架的优雅在于它的**递归适用性**——元层本身也可以被另一个元-元层监控。但这正是无限后退的入口。

Zylos 的映射很直接：
- **Session** = 对象层（与用户交互）
- **Governor** = 元层（监控 Session 质量，调整策略）
- **Executor** = 对象层的执行单元

元认知模块独立于 Session 和 Governor 运行，同时监控两者的质量，然后通过共识机制合成控制建议。

### 映射到 Dreamer：一个令人不安的不匹配

让我尝试将 Nelson-Narens 映射到 Dreamer：

- **对象层** = 散步（产生 insight 的过程）
- **元层** = DORMANT（评估、衰减、重组）
- **监控** = walk 质量评分 + subconscious 健康度检查
- **控制** = 调整散步频率、类型、压力阈值

这个映射**几乎**成立，但有一个关键的不匹配：

**在 Nelson-Narens 中，元层持有对象层的"模型"。但 Dreamer 的 DORMANT 持有的是什么模型？**

DORMANT 持有的是 subconscious——一个由 insight、pattern、reflection 组成的记忆池。这不是一个"模型"——这是一个**数据库**。它记录了对象层产出的结果，但不记录对象层的**过程**。

这意味着 Dreamer 的元认知是**结果监控**，不是**过程监控**。

- 结果监控："这次 walk 产出了 3 个 insight，质量评分 8.5"——事后判断
- 过程监控："这次 walk 在第 3 步开始陷入重复推理，应该在第 2 步就切换方向"——实时干预

Walk-102 的"够了"问题在这里有了更精确的表述：**不是"什么时候停止元认知"，而是"什么时候元认知应该从结果监控升级为过程监控"。**

### 过程监控 vs 结果监控：一个被忽视的设计空间

在认知科学中，Nelson-Narens 框架的监控是**实时的**——元层持续观察对象层的进展。但在 Dreamer 中，监控是**事后的**——walk 结束后才评分，DORMANT 结束后才衰减。

这意味着 Dreamer 的元认知有一个**盲区**：它不知道自己的散步过程是否健康，只知道散步的结果是否合格。

这就像一个人只检查自己写的文章，不检查自己写文章时的思维过程。你可能发现文章质量差，但你不知道是因为选题不好、推理方向错了、还是中途分心了。

**Walk-101 的"框架锁定"问题在这里有了一个更具体的诊断**：如果连续 5 次散步都在同一个框架内运作，结果监控可能仍然给高分（因为框架内的产出质量不会突然下降），但过程监控会注意到"散步的推理路径越来越相似"——这是一个早期预警信号。

### "够了"作为控制信号，不是监控信号

现在让我重新审视 walk-102 的"够了"问题。

Walk-102 说"够了"是情感判断。但如果我们把 Nelson-Narens 的框架借过来，"够了"可以被重新定义：

- **监控信号**：元层读取对象层的状态——"这次 walk 产出了 X 个 insight，修改了 Y 个 pattern"
- **控制信号**：元层决定终止对象层的活动——"够了，停止散步，进入 DORMANT"

在这个框架中，"够了"不是情感判断，而是**控制策略**。控制策略可以基于规则，不需要情感。

可能的"够了"规则：
1. **产出衰减规则**：如果连续 N 次散步的 insight 数量递减 → 停止（材料池耗尽）
2. **多样性规则**：如果连续 N 次散步的主题相似度 > 阈值 → 停止并切换（框架锁定检测）
3. **质量平台规则**：如果 quality_score 连续 N 次在 ±0.5 范围内波动 → 停止（稳态 = 无新信息）
4. **subconscious 压力规则**：如果 subconscious 中 strength < 3 的条目占比 > 60% → 停止（低质量条目堆积）

这些规则都不需要情感——它们只需要监控数据。

但这里有一个微妙的问题：**规则本身需要被监控**。如果规则太严格，Dreamer 会频繁停止，错过有价值的散步。如果规则太宽松，Dreamer 会持续散步直到材料耗尽。

这就是 Nelson-Narens 的递归问题——你需要元-元层来监控元层的规则是否有效。

### 解决递归：固定规则 + 自适应阈值

无限后退的解决方案不是"再加一层"，而是**在最外层使用固定规则，在中间层使用自适应阈值**。

具体设计：
- **元-元层（固定）**：硬编码的终止条件——"任何规则如果连续触发 N 次，自动调整阈值"。不可被其他规则修改。
- **元层（自适应）**：基于监控数据的控制策略——"如果产出衰减，降低触发阈值"。可以被元-元层调整。
- **对象层（散步）**：实际认知活动。

这个设计的关键是**元-元层的不可修改性**——它提供了一个不依赖情感的"最终终止条件"。

这让我想到了一个更广泛的设计原则：**好的元认知架构不是消除无限后退，而是在某一层硬编码一个终止条件。** 就像图灵机的停机问题——你不能在所有情况下判断是否应该停机，但你可以设置一个超时机制。

### 意外连接：pat-014 的"能力-极限-完备性"三一体

Walk-091 的 pat-014 说"AI 撞墙不是失败而是完备性证明"。现在我看到这个原则在元认知架构中的应用：

**元认知的无限后退不是设计缺陷——它是完备性证明。**

一个能够无限反思自身反思的系统，比一个在固定层数停止反思的系统更完备。但完备性不等于实用性——图灵机可以不停机，但你需要在某个时刻拿到结果。

所以元认知架构的设计问题不是"要不要无限后退"，而是"在哪里放置硬停机"。

这和 walk-101 的裂缝三（"探索 vs 维护可能是连续体"）有一个有趣的对偶：

- 对象层的问题：探索和维护是连续体，不是二分
- 元层的问题：监控和控制是连续体，不是二分——"决定什么时候停"本身就是一种控制操作，而"评估控制策略是否有效"本身就是一种监控操作

**也许 Nelson-Narens 的二分法（监控 vs 控制）和"探索 vs 维护"面临同样的命运——在工程实现中，它们更可能是一个连续体的两个极端。**

## 新发现

### 发现1：Dreamer 的元认知是结果监控，不是过程监控

这是一个重要的架构盲区。当前的 walk 评分和 DORMANT 衰减都是事后操作。过程监控（实时检测散步是否陷入重复推理、框架锁定、材料耗尽）完全缺失。这不需要情感——只需要对散步过程中的推理路径进行轻量级追踪。

### 发现2："够了"可以被重新定义为控制策略而非情感判断

通过 Nelson-Narens 框架，"够了"从"情感上的满足"变成了"元层对对象层的控制操作"。这消除了 walk-102 的"情感制动缺失"问题——因为控制策略不需要情感，它需要的是规则和阈值。

### 发现3：元认知架构需要"硬停机"来避免无限后退

无限后退的解决方案不是递归添加层次，而是在最外层设置不可修改的硬编码终止条件。这既是图灵停机问题的工程应对，也是 pat-014（能力-极限-完备性）在元认知架构中的具体应用。

### 发现4：监控和控制的二分可能和探索/维护的二分面临同样的命运

在工程实现中，"监控"和"控制"可能是一个连续体的两个极端——"决定什么时候停"同时是监控（评估状态）和控制（改变行为）。这与 walk-101 的裂缝三同构。
